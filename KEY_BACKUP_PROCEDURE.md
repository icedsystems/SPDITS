# FIELD_ENCRYPTION_KEY — Backup & Recovery Procedure

**Classification: CONFIDENTIAL — Internal Use Only**
**System: ICED SPDITS**
**Last reviewed: May 2026**

---

## What this key does

`FIELD_ENCRYPTION_KEY` is a Fernet symmetric encryption key that protects all direct
participant identifiers stored in the database:

- Full name
- National ID / Passport number
- Phone number / Alternative phone
- Email address
- Physical address

Without this key, **all IdentityMap records in the database are permanently unreadable**.
Losing the key means losing the ability to re-identify any participant — ever.

---

## Key format

The key is a 32-byte URL-safe base64-encoded string generated by the Fernet library.
It looks like this (example — NOT a real key):

```
dGhpcyBpcyBhIGZha2Uga2V5IGZvciBkb2N1bWVudGF0aW9u
```

Length: exactly 44 characters ending with `=`

## Backup procedure

### Step 1 — Read the key from the server

SSH into the server and print only the key line:

```bash
ssh root@kenya1
grep FIELD_ENCRYPTION_KEY /home/ipo/spdits/.env
```

You will see:
```
FIELD_ENCRYPTION_KEY=<your-key-here>
```

Copy the value (everything after the `=` sign).

### Step 2 — Verify the key is working before backing up

Run the verification script from inside the container:

```bash
docker compose exec web python manage.py shell -c "
from apps.participants.models import get_fernet, IdentityMap
f = get_fernet()
test = f.encrypt(b'verification-test')
result = f.decrypt(test)
print('KEY OK:', result == b'verification-test')
count = IdentityMap.objects.count()
print('IdentityMap records protected by this key:', count)
"
```

Expected output:
```
KEY OK: True
IdentityMap records protected by this key: <number>
```

If you see `KEY OK: False` or an exception — stop. Do not proceed until resolved.

### Step 3 — Store the key in at least TWO secure locations

Choose from these options (use at least two):

#### Option A — Password manager (recommended primary)
Store in a password manager (e.g. Bitwarden, 1Password, LastPass) under:
- **Title:** `ICED SPDITS — FIELD_ENCRYPTION_KEY (Production)`
- **Username:** `spdits-production`
- **Password:** `<the key value>`
- **Notes:** Include the server hostname, date backed up, and who backed it up

#### Option B — Encrypted file on secure offline storage
```bash
# On your local machine — create an encrypted backup file
echo "FIELD_ENCRYPTION_KEY=<your-key>" > spdits_encryption_key_backup.txt
gpg --symmetric --cipher-algo AES256 spdits_encryption_key_backup.txt
# This creates spdits_encryption_key_backup.txt.gpg
# Store the .gpg file on a USB drive kept in a locked physical location
# DELETE the plaintext .txt file after encrypting
rm spdits_encryption_key_backup.txt
```

#### Option C — Secure note in an institutional secrets vault
If ICED uses Azure Key Vault, AWS Secrets Manager, or similar — store it there
under the name `spdits-prod-field-encryption-key`.

### Step 4 — Record the backup

Fill in and keep this log:

```
Date backed up:       _______________
Backed up by:         _______________
Backup locations:     1. _______________
                      2. _______________
Key fingerprint*:     _______________
IdentityMap count:    _______________
```

*Key fingerprint (run on server — does not reveal the key):*
```bash
grep FIELD_ENCRYPTION_KEY /home/ipo/spdits/.env | cut -d= -f2 | sha256sum
```
Record only the SHA-256 hash (not the key itself) as a fingerprint to verify future copies match.

## Recovery procedure

If the server is lost and you need to restore:

1. Restore the `.env` file with the backed-up key value
2. Restore the database (PostgreSQL dump)
3. Start the containers: `docker compose up --build -d`
4. Run verification script (Step 2 above) to confirm decryption works
5. Run migrations: `docker compose exec web python manage.py migrate --noinput`

**Both the database dump AND the key are required to recover participant identity data.**
Storing them in the same place defeats the purpose — keep them separately.

---

## Key rotation procedure

Only rotate the key if you suspect it has been compromised.
**Rotation requires a migration script — do not just replace the key in `.env`.**

Steps:
1. Generate a new key:
   ```bash
   docker compose exec web python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
   ```
2. Run the rotation migration script (re-encrypts all IdentityMap records):
   ```bash
   docker compose exec web python manage.py rotate_encryption_key --old-key <old> --new-key <new>
   ```
   *(This script must be written before rotation — contact the system developer.)*
3. Update `/home/ipo/spdits/.env` with the new key
4. Rebuild containers: `docker compose up --build -d`
5. Verify with the verification script
6. Back up the new key following the procedure above
7. Securely destroy all copies of the old key
