1. System Overview

ICED SPDITS (Secure Partner Data Intake & Tracing System) is a web-based platform that manages the full lifecycle of participant data from initial upload through field interviews. It is designed for data security and auditability, with strict role-based access controls.

User Roles — six distinct roles with separate access levels
RoleWho They AreWhat They Can Do
System Admin ICED system administrators Full access — manage all users, approve uploads, view all data, access audit logs, re-identify participants, configure SFTP and email settings
Partner Data partner organisations (NGOs, agencies) Upload participant CSV/XLSX files, view their own upload history and batch status
Supervisor Field team supervisors Assign traced participants to their enumerators, monitor team progress, update interview status on behalf of enumerators, manage their enumerators' accounts
Enumerator Field data collectors View their assigned participants, record interview outcomes (Interviewed / Refused / Unreachable / Callback), download their secure assignment CSV
Tracer Participant tracers Work through the tracing queue, update tracing status (tracing / traced / cannot trace), add location and contact notes
Compliance Officer Data compliance staff Access the compliance dashboard, re-identify participants with mandatory reason logging, view compliance reports

Pseudocode system: When a batch is approved, each participant is assigned a unique pseudocode (format: PSN-YYYY-NNNNNN). Direct identifiers (name, phone, ID) are stripped and stored encrypted. The main participant record only holds quasi-identifiers (county, gender, age). This protects participant privacy throughout the workflow.

3. Partner Role — Navigation & Workflow

Partners are external organisations that supply participant data to ICED. They can only see their own data.

What Partners See
PagePathDescription
Dashboard/dashboard/Their own KPIs: total uploads, pending / approved / rejected batch counts, list of recent batches
File Uploads/uploads/Upload a new CSV or XLSX file. View their own upload history and current batch statuses.
Upload File Requirements
  • Accepted formats: CSV or XLSX
  • Required columns: name, national_id or passport, phone, county, gender, age
  • Files are stored under /media/uploads/<PARTNER_CODE>/ on the server
  • Duplicate detection runs on national ID / passport number. Duplicate rows are flagged but the rest of the batch is still processed.
Note for admins: When creating a partner user account, always link them to the correct Partner organisation in the user form. Without this link, they cannot upload files.

7. Compliance Officer — Navigation & Workflow

The Compliance Officer has a restricted but sensitive role — they can access the re-identification feature to view a participant's real identity when operationally necessary.

PagePathDescription
Compliance Dashboard/compliance/Summary of re-identification events, participant counts, compliance KPIs
Re-Identification/participants/<id>/reidentify/View a participant's decrypted direct identifiers. Requires a mandatory written reason. Decrypted data is displayed for 60 seconds then auto-hidden. Every access is logged immutably.
Participants/participants/Read-only access to the full participant list (quasi-identifiers only)
Audit Log/audit/Access to the full audit trail
Re-identification protocol: Every re-identification access is logged with: the officer's username, the participant pseudocode, the stated reason, the IP address, and the session ID. These logs cannot be deleted. Admins can review all re-identification events from the Audit Log.

10. Tracing Workflow

Once a batch is approved and processed, participants appear in the tracing queue with status Uploaded. Tracers work through this queue.

  1. Tracer goes to /tracing/queue/ and opens a participant
  2. Updates status to Tracing while actively searching
  3. Once located, marks as Traced and fills in: contact method, location found, and any tracing notes
  4. A TracingLog entry is created for every status change — full history is kept
  5. Traced participants move to the traced queue and become available for supervisor assignment
Only participants with status Traced appear in the assignment queue. If a participant disappears from the assignment queue, check their tracing status — they may still be in the tracing queue.
11. Assignment Workflow
  1. Supervisor goes to /assignments/queue/
  2. Selects one or more participants using the checkboxes
  3. Selects an enumerator from the dropdown (supervisors only see their own enumerators)
  4. Clicks Bulk Assign
  5. System creates Assignment records, updates participant status to Assigned, generates a secure CSV with decrypted participant identity data, and emails the enumerator a 30-minute download link
  6. The enumerator also receives an in-app notification
Re-assigning: If a download link expires before the enumerator downloads it, the supervisor can re-assign the same participants to regenerate a fresh link.
Secure CSV: The download CSV contains the enumerator's participants' full decrypted identities (name, phone, address, ID). It expires after 30 minutes. This ensures PII is never stored in plaintext on the server beyond what is necessary.
12. Interview Workflow
  1. Enumerator goes to /assignments/mine/
  2. Each row has a Record Outcome dropdown. They choose the result of their field visit.
  3. A modal confirms the choice, allows notes and a callback date if needed
  4. An Interview record is created (or updated if already exists). Every change is recorded in InterviewStatusHistory.
  5. On Interviewed: participant status updated to Interviewed, assignment marked Completed
  6. On Refused or Unreachable: assignment also marked Completed
Supervisor Override

Supervisors can update any enumerator's participant outcome from /assignments/ (Team Assignments). This is designed for when an enumerator has no internet access. The audit log records the supervisor as the person who made the change.

13. SFTP Ingestion

SFTP ingestion allows partners to drop files into a secure FTP folder rather than uploading through the web interface. The system polls every 60 seconds.

Configuring SFTP for a Partner
  1. Go to /sftp/config/ and click Add SFTP Config
  2. Enter the host, port, username, and password for the partner's SFTP account
  3. Set the Remote Folder — the folder the system will watch for new files
  4. Link to the correct Partner organisation
  5. Toggle Active to enable polling
Monitoring SFTP Activity

All ingestion events are logged at /sftp/. Each log entry shows the detected file, its status (pending / processing / completed / failed), and any error messages. Failed ingestions do not affect other files.

SFTP keys must be generated before the first server start. Run bash scripts/generate_sftp_keys.sh from the project root. If the SFTP container fails to start, this is the most common cause.
14. Audit Log & Security Controls
Audit Log

Every significant action in the system creates an immutable AuditLog entry. These records cannot be edited or deleted. Each entry captures:

  • Action type — e.g. LOGIN, UPLOAD, APPROVE_BATCH, INTERVIEW_UPDATE, REIDENTIFY
  • User — who performed the action
  • Timestamp — exact date and time (Africa/Nairobi timezone)
  • IP address — originating IP
  • Session ID — to correlate with other actions in the same session
  • Old and new values — for status changes
  • Description — human-readable summary
Session Management
  • Sessions expire after a configurable period of inactivity (default: 30 minutes). The user is redirected to login with a "session expired" message.
  • When a new browser tab is opened while someone is already signed in, a session confirmation modal appears asking to continue or switch user.
  • Only one active Django session per browser. Logging out one tab logs out all tabs in that browser.
Data Protection
  • Fernet encryption — all direct identifiers (name, ID, phone) are encrypted at rest with a symmetric key (FIELD_ENCRYPTION_KEY). If this key is lost, direct identifiers cannot be recovered.
  • Pseudocode separation — the main participant record only stores quasi-identifiers. The identity is in a separate IdentityMap table.
  • Re-identification gate — only System Admins and Compliance Officers can access the re-identify view. Every access is logged.
  • HTTPS + mTLS — all browser traffic is encrypted in transit via Nginx + mutual TLS.
15. Common Issues & Troubleshooting
ProblemLikely CauseSolution
User receives welcome email but the link is expired Password setup links expire after 24 hours Admin goes to User Management, finds the user, clicks Reset Password to send a new link
Cannot create a new user — "Email already exists" error An account with that email is already registered Search the user list for the email. If active, no new account needed. If inactive, re-enable the existing account.
Participant not appearing in Assignment Queue Participant is not in Traced status Check the Tracing Queue — the participant may still be in Tracing or Cannot Trace status
Enumerator says download CSV link doesn't work 30-minute token has expired Supervisor re-assigns the participants to generate a fresh link and email
Upload file fails validation Missing required columns or data format errors View the batch detail for specific row-level errors. Partner should correct the file and re-upload.
SFTP ingestion shows "Failed" entries Connection error, wrong credentials, or malformed file Go to SFTP Config, verify host/port/credentials. Check the SFTP ingestion log for the specific error message.
User sees blank page after login Account has no role assigned Go to User Management, find the user, edit and assign the correct role
Session confirmation prompt doesn't disappear Browser storage issue Clear browser site data / cookies and re-login. The prompt will appear once on the next new tab.
Server-level issues (database down, Celery not processing, email not sending) require access to the production server. Contact the system administrator at root@kenya1. All application logs are available via docker compose logs -f.
